## Acceptance Rules

- `Source` identifies where the screenshot came from.
- `Provider` identifies who supplied it or which project published it.
- `Permission` states why public reuse in this repository is allowed.
- `Permission evidence` points to an explicit written approval, public license, or source page.
- `Screenshot file` names the exact repository path used for the README image.
- `Public README use approved` is `yes` only after source and permission are clear.
- `Sensitive content review` confirms no private chats, email addresses, account names, tokens, API keys, billing data, usage data, internal hostnames, or private project names are visible.
- `Visual match review` confirms the image represents behavior this repository can plausibly produce.
- `Crop or redaction applied` records any edits made before publication.
- `Runtime boundary` states that the image was not produced in this Windows workspace and has not been locally re-verified with `verify-signal-garden-skin.sh`.
- `README wording` must describe it as a community-provided runtime reference screenshot, not as local verification evidence.
- `Reviewer` and `Review date` identify who accepted the image and when.

## Rejection Rules

Do not publish the screenshot if any item below is true.

- Public reuse permission is missing or ambiguous.
- The image shows private conversations, credentials, API keys, account identifiers, payment or billing details, usage data, internal domains, or private project names.
- The image is unrelated to Signal Garden or shows an effect the current code cannot plausibly produce.
- The image is edited so heavily that it becomes a mockup rather than a runtime reference.
- README wording would imply the screenshot was generated by this repository's verification flow when it was not.
